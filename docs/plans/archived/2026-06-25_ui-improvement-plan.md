## Goal

根據實際打開的 otter 畫面，規劃並實作一組小而可驗證的介面改善，優先解決目前最影響使用的版面與資訊層級問題。

## Context

已用 dev compose 啟動 app，登入 `admin@otter.local`，建立含 3 位參與者與 3 筆支出的旅行，並在 Chrome 實際檢視目前桌面與手機寬度畫面。

觀察到的主要問題：

1. 左側「旅行 / 群組」卡片被右側內容撐到約 2467px 高，表單與旅行列表被拉開，出現大量空白；這是最明顯的版面 bug。
2. 旅行標題列同時塞入匯出、列印、改貨幣、重新命名、刪除等動作，按鈕換行後層級混在一起，危險動作不夠分離。
3. 核心結果「分帳結果 / 結清建議」位於整頁底部，要先經過參與者、記帳表單與支出列表，使用者很晚才看到最重要的結論。
4. 每筆支出有 7 個操作按鈕，列表可讀性被操作區吃掉；常看資訊與少用操作沒有分層。
5. 新增旅行表單一直佔據左欄上方；已有旅行時，瀏覽與切換旅行比新增旅行更常用。

## Non-Goals

- 不改拆帳規則、資料模型、API 或權限。
- 不引入 UI framework 或新依賴。
- 不把現有 `prompt()` / `confirm()` 流程改成大型 modal 系統。

## Plan

- [x] 修正 dashboard 外層 grid 的 stretch 問題，讓左側旅行卡片內容靠上排列；以 Chrome 實際檢視確認左欄不再出現大段空白，並用 `npm run check` 驗證。
- [x] 將左側旅行新增表單收合到原生 `<details>`，預設顯示旅行列表；以 Chrome 確認已登入且已有旅行時，切換旅行不必先越過新增表單。
- [x] 重整旅行標題卡：標題與基準貨幣保留在第一層，匯出 / 列印放同一操作群，重新命名 / 改基準貨幣放次要操作群，刪除旅行獨立靠後；以 Chrome 確認危險動作與一般動作分開。
- [x] 將「分帳結果 / 結清建議」提升到新增支出表單之前，讓使用者選旅行後先看到目前誰該付誰；以 Chrome 截圖確認結果在首屏或接近首屏內可見。
- [x] 將支出列表每筆的少用編輯動作放進原生 `<details>`「更多操作」區，只保留支出摘要與刪除可快速掃描；以 `src/client/views.test.ts` 補 selector/ARIA smoke test，並跑 `npm test -- src/client/views.test.ts`。
- [x] 做一次 620px 以下的手機寬度檢查，確認卡片順序為：旅行切換、結果摘要、新增支出、支出列表；以 Chrome viewport 390px 檢查與截圖作為驗收證據。

## Risks

- 移動結果區會改變既有瀏覽順序；但結果是拆帳 app 的核心輸出，提前顯示更符合使用者目標。
- 收合新增旅行 / 支出操作可能讓新手少一步發現功能；使用原生 `<summary>` 明確標示可降低風險。
- 支出操作收合後，事件 selector 必須保持穩定；用既有 view smoke test 覆蓋。
